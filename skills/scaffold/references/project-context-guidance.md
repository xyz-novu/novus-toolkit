# Project Context — Writing Guidance

How to populate `project-context.md` well. Read this when writing the project context during scaffold creation.

## Principles

- **Plain language.** No consultant-speak. "The client wants faster onboarding" not "The client seeks to optimise their onboarding velocity."
- **Specific over generic.** "6-week engagement, Jan–March" not "medium-term engagement."
- **Honest gaps.** If something is unknown at scaffold time, leave the field present but note "TBD". Don't fabricate detail.
- **The file grows.** Project-context is a living artifact. The scaffold creates the structure and seeds it with what's known. The user fills it in as the project progresses.
- **Only write what was said.** Populate from Q&A data. A sparse but accurate project context is more useful than a rich but speculative one.

## Section-by-Section Guidance

### Engagement Overview

The executive summary. Every field should be populated from Q&A data.

- **Purpose:** 1–2 sentences capturing the core goal. Make it concrete.
  - Good: "Identify critical friction points in the journey from initial engagement to active product use across IATA platforms."
  - Bad: "Improve the customer experience."
- **Strategic context:** The "why behind the why". Relationship history, business drivers, what makes this engagement different. This is where nuance lives.
  - Good: "This is the 4th engagement with IATA. Previously smaller scoped but strategic. The company has no proper product processes. This project is about finally being customer-focused, and has received buy-in across the business."
  - Bad: "The client wants to improve their processes and we're helping them."
- **Phase:** Set to the starting activity from the Q&A (e.g. "Research — stakeholder interviews" or "Design — initial specs"). Update this field as the project progresses.

### Scope & Deliverables

Adapt sub-headings to the project type:

- Research-heavy → "Research scope" and "Key outputs"
- Design-only → "Design scope"
- Strategy → "Strategic scope" and "Expected outcomes"
- Process/methodology → "Scope" and "Key outputs"

Start with what's known. It's fine if deliverables are tentative at scaffold time. Better to write "TBD — to be defined after initial research" than to invent a list.

### Key Contacts

The Notes column is the most valuable part — it captures context that makes interactions smoother. Populate from Q&A data; leave blank if unknown rather than inventing.

Good notes:
- "Owns the customer portal. Pushes for quantification. Day-to-day decision-maker."
- "Platform architecture, Salesforce, analytics direction."
- "20 years at IATA. Institutional memory. Revenue context."

Bad notes:
- "Important stakeholder."
- "Will be involved in the project."

If a person was mentioned but their role is unknown, include them with "TBD" in the Role column rather than guessing.

### Key Dates

Start with known dates. Use date ranges where appropriate. The Status column tracks progress — `Planned`, `In progress`, `Complete`.

### Working Notes

Running context. Not formal enough for the sections above. Good candidates:

- Alignment signals from the setup conversation
- Political considerations
- Emerging hypotheses
- Tooling decisions already made
- Things the user mentioned they'd supply later

## Tone

Write as if a colleague is reading this for the first time to get up to speed. Give them the real picture — the messy dynamics, the unstated tensions, the things that aren't in the formal brief. That's what makes project-context useful beyond the first week.
